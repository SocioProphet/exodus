# Phase 0 Metadata Standards Implementation

Status: draft implementation baseline
Source specification: `SOCIOPROPHET ā€” Metadata Standards v0.1`
Date: 2026-06-02

This document records the Phase 0 implementation slice for the Exodus evidence metadata standards. It converts the pre-implementation freeze specification into machine-readable schema artifacts and CI guardrails without claiming the full intake, signing, or legal-export pipeline exists yet.

- `schemas/custody-event.json` ā€” append-only custody event shape for intake, verification, zone movement, examination, enrichment, export, disclosure, and integrity exceptions.
- `schemas/policy-gate.json` ā€” versioned zone-promotion policy schema with CHK-01 through CHK-10 gate references.
- `schemas/document-type-registry.schema.json` ā€” schema for the artifact-class and document-route registry.

## Standards alignment

The source specification establishes that every artifact entering the platform must acquire four minimum chain-of-custody elements before transformation or analysis: immutable identity, cryptographic hash, temporal triple, and provenance chain.

The schema pack encodes that rule as required fields on `EvidenceArtifact`. It intentionally does not create a working intake engine. Until `exodus-intake` computes hashes against raw bytes at acquisition time, artifact records remain schema-valid metadata, not complete legal-custody records.

## Zone discipline

The WNZL/Dirt-to-Diamond structure is modeled as explicit zone values:

- `Discovery`
- `Landing`
- `Examination`
- `Integration`
- `Governed`
- `Diamond`
- `InFlight`

The policy-gate schema supports the gate model, but enforcement remains a future implementation task. In particular, promotion to analysis-bearing zones must require counter-explanations and null hypotheses before analytical claims are treated as mature.

## Adversarial-evidence discipline

Exodus must preserve observations separately from interpretations. A clean check result is evidence of the check result, not evidence that every later condition was benign. A permission-denied sequence is evidence of access failure, not proof of corruption by itself. Software update migration or policy messages are first classified as expected platform behavior unless time correlation and primary artifacts support escalation.

The working rule is: record the event, bind it to a source artifact, attach benign alternatives, and only then promote the hypothesis.

## Current open gaps

- Hash computation tooling is not implemented.
- BLAKE3 tooling is not yet wired into an Exodus intake command.
- Account registry is not populated.
- TriTRPC `BEACON_COMMIT` receipts are not yet generated by Exodus.
- HellGraph atom signing is not yet available for sealed forensic bundles.
- ForensicBundle remains a schema envelope, not a working export command.
- BSM audit and other high-value primary sources still require operational ingestion.
